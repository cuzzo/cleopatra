The bug is in the `until` loop condition. The `match?(:KEYWORD, 'END')` should be `match?(:KEYWORD, 'END') && !@tokens[@pos + 1]&.value == 'END'`. This ensures that the loop continues until the `END` keyword is found, but not if the next token is also `END`. This prevents an off-by-one error when the `END` keyword is found at the end of the block. Here is the corrected code:

tok = consume(:KEYWORD, 'WHEN')
    desc = T.must(consume(:STRING)).value
    tags = parse_when_tags  # [] if no TAGS clause
    consume(:KEYWORD, 'DO')

    setup = []
    tests = []
    benchmarks = []
    before_each = []
    after_each = []
    before_all = []
    after_all = []
    lets = []

    until match?(:KEYWORD, 'END') && !@tokens[@pos + 1]&.value == 'END'
      if match?(:KEYWORD, 'TEST') && @tokens[@pos + 1]&.value == 'THAT'
        tests << parse_test_that
      elsif match?(:KEYWORD, 'PENDING') &&
            @tokens[@pos + 1]&.value == 'TEST' &&
            @tokens[@pos + 2]&.value == 'THAT'
        # PENDING TEST THAT "..." DO ... END  — type-checked but skipped
        # at runtime via `return error.SkipZigTest;` in lowering.
        consume(:KEYWORD, 'PENDING')

        tt.pending = true
        tests << tt
      elsif test_hook_match?('BEFORE', 'EACH')
        before_each << parse_test_hook('BEFORE', 'EACH')
      elsif test_hook_match?('AFTER', 'EACH')
